# Auto-update

Release builds of o3 update themselves through the platform's native update
framework: [Sparkle](https://sparkle-project.org) on macOS and
[WinSparkle](https://winsparkle.org) on Windows. Linux (AppImage) and dev
builds keep the built-in check-only flow: o3 notifies about a new release and
opens the download in the browser.

## How it works

| | macOS | Windows | Linux / dev builds |
| --- | --- | --- | --- |
| Framework | Sparkle 2 via [go-sparkle](https://github.com/abemedia/go-sparkle) | WinSparkle via [go-winsparkle](https://github.com/abemedia/go-winsparkle) | none (custom flow) |
| Feed | `releases/latest/download/appcast.xml` | same | GitHub Releases API |
| Verifies | EdDSA signature on the `.dmg` | EdDSA signature on the `setup.exe` | n/a (browser download) |
| Installs | in place, relaunches | quits o3, runs the installer silently | user installs manually |

The pieces:

- **Build tag `native_updater`** — set only by release CI and
  `make … NATIVE_UPDATER=1`. Without it (`wails dev`, `go test`, all Linux
  builds) `newNativeUpdater()` returns nil and nothing framework-related is
  even linked. `AppInfo.updateMode` tells the frontend which mode is running.
- **`updater_native_sparkle.go` / `updater_native_winsparkle.go`** — thin
  adapters: start the framework with the persisted auto-check pref, forward
  the explicit "Check for Updates" gesture, mirror the Settings toggle. The
  frameworks own everything else (dialogs, download, verify, install,
  relaunch, skipped-version bookkeeping).
- **`scripts/sparkle-framework.sh`** — embeds the pinned, SHA-256-verified
  Sparkle.framework into `o3.app/Contents/Frameworks` after `wails build`
  (the binary links go-sparkle's stub and resolves the real framework via an
  `@loader_path/../Frameworks` rpath).
- **`scripts/appcast.mjs`** — generates the shared feed: one `<item>` per
  platform, discriminated by `sparkle:os`. `sparkle:version` is the numeric
  `X.Y.Z` because that is what installed apps report (`CFBundleVersion` on
  macOS, `update.Numeric(version)` registered with WinSparkle).
- **`appcast` job in `.github/workflows/release.yml`** — signs the `.dmg` and
  `setup.exe` with Sparkle's `sign_update`, generates `appcast.xml`, and
  attaches it to the same **draft** release as the binaries. Publishing the
  release therefore exposes the feed and its enclosures atomically;
  `/releases/latest/` never serves drafts or prereleases.

The builds are not code-signed, so the EdDSA signature is the **only**
integrity check on an update. That is why both the packaging script and the
release workflow refuse to proceed while the key placeholders are empty, and
why the WinSparkle adapter disables itself rather than start without a key.

## One-time key setup (maintainer)

Updates are signed with an ed25519 keypair generated by Sparkle's tooling.
**Losing the private key strands every installed user** — a new key cannot
sign updates that existing installs will accept — so treat it like a signing
certificate.

1. Download and verify the pinned Sparkle release (same version as
   `scripts/sparkle-framework.sh`; check its `SPARKLE_URL`/`SPARKLE_SHA`),
   then extract it.
2. Generate the keypair (the private key lands in your login Keychain):

   ```sh
   ./bin/generate_keys
   ```

   It prints the **public** key (base64).
3. Export the private key and store it as the repo secret:

   ```sh
   ./bin/generate_keys -x sparkle_ed_private_key
   gh secret set SPARKLE_ED_PRIVATE_KEY --repo AngelMsger/o3 < sparkle_ed_private_key
   ```

4. Back the exported file up in a password manager, then delete it.
5. Commit the **public** key in the two places that pin it (they must match):
   - `build/darwin/Info.plist` → `SUPublicEDKey`
   - `updater_native.go` → `sparkleEdPublicKey`

## Local end-to-end test (macOS)

Never test with the production key — generate a throwaway pair
(`./bin/generate_keys` under a scratch `$HOME`) and put its public key in the
two pinned locations on a scratch branch.

1. Build and "install" the old version:

   ```sh
   make dmg VERSION=0.1.0 NATIVE_UPDATER=1
   cp -R build/bin/o3.app /Applications/
   ```

2. Build the new version and sign its dmg:

   ```sh
   make dmg VERSION=0.2.0 NATIVE_UPDATER=1
   ./bin/sign_update --ed-key-file test.key -p build/bin/o3-0.2.0-universal.dmg
   ```

3. Generate a local appcast and serve it:

   ```sh
   node scripts/appcast.mjs --version 0.2.0 --numeric-version 0.2.0 --tag v0.2.0 \
     --base-url http://localhost:8000 \
     --dmg build/bin/o3-0.2.0-universal.dmg --dmg-sig '<sig from step 2>' \
     --setup <any placeholder .exe> --setup-sig placeholder \
     --out serve/appcast.xml
   cp build/bin/o3-0.2.0-universal.dmg serve/ && (cd serve && python3 -m http.server 8000)
   ```

4. Launch 0.1.0 against the local feed and drive the update:

   ```sh
   O3_APPCAST_URL=http://localhost:8000/appcast.xml /Applications/o3.app/Contents/MacOS/o3
   ```

   Help → **Check for Updates…** should show Sparkle's dialog; **Install and
   Relaunch** should land you in 0.2.0. Also worth checking: the Settings
   toggle flips `defaults read com.angelmsger.o3 SUEnableAutomaticChecks`, a
   tampered dmg is rejected, and plain `wails dev` still shows the custom
   in-app update sheet.

Windows smoke test: same idea in a VM — install `0.1.0` from a
`native_updater` setup.exe, serve a signed `0.2.0` appcast from the host, set
`O3_APPCAST_URL`, and confirm WinSparkle's dialog appears, o3 quits, the
installer runs, and a tampered installer is rejected.

## Caveats

- **Prereleases**: an installed `1.2.3-rc.1` reports the numeric `1.2.3`, so
  it is never offered the final `1.2.3`. RC distribution is manual.
- **Portable zip (Windows)**: WinSparkle serves the installer enclosure to
  portable users too — accepting an update converts a portable copy into an
  installed one.
- **Sparkle 3**: go-sparkle drives Sparkle through the `SUUpdater`
  compatibility shim, which Sparkle 2.x keeps but Sparkle 3 will remove.
  Revisit the binding before bumping the pin past 2.x.
